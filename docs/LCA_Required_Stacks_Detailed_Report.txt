STACK 2: CONNECTINTEGRATIONSTACK  (LCA-CONNECTINTEGRATIONSTACK-9RNPIXIVT080)
================================================================================

PURPOSE:
  Integrates Amazon Connect (your phone/contact center system) with the
  LCA pipeline. Listens for call events from Connect and routes them into
  the Kinesis stream so LCA can process them.

TOTAL RESOURCES: 8

--------------------------------------------------------------------------------
RESOURCE LIST & FUNCTIONALITY
--------------------------------------------------------------------------------

1.  ContactEventProcessorFunction  (AWS::Lambda::Function)
    - The bridge between Amazon Connect and LCA.
    - Triggered by EventBridge when a call starts or ends in Connect.
    - Reads contact data (call ID, agent ID, phone numbers, timestamps).
    - Writes a call start/end event into the LCA Kinesis Data Stream.
    - This is what makes a call appear in the LCA UI when it begins.

2.  ContactEventProcessorFunctionRole  (AWS::IAM::Role)
    - IAM role for ContactEventProcessorFunction.
    - Permissions: read Connect contact data, write to Kinesis stream,
      write CloudWatch logs.

3.  EventBridgeRuleToTriggerContactEventFunction  (AWS::Events::Rule)
    - An EventBridge rule that watches for Amazon Connect contact events.
    - Events monitored: INITIATED (call starts), CONNECTED, DISCONNECTED
      (call ends).
    - Automatically triggers ContactEventProcessorFunction when these
      events occur.

4.  AllowEventBridgeToCallContactEventFunction  (AWS::Lambda::Permission)
    - Grants EventBridge permission to invoke ContactEventProcessorFunction.
    - Security boundary — only EventBridge can trigger this Lambda,
      not other services.

5.  AssociateInstanceFunction  (AWS::Lambda::Function)
    - Runs once during deployment to associate your Amazon Connect
      instance with the LCA stack.
    - Links the Connect instance so EventBridge can receive its events.

6.  AssociateInstanceFunctionResource  (AWS::CloudFormation::CustomResource)
    - CloudFormation custom resource that invokes AssociateInstanceFunction
      during stack deployment.
    - Ensures the Connect association is set up before the stack finishes.

7.  LambdaRole  (AWS::IAM::Role)
    - IAM role for AssociateInstanceFunction.
    - Permissions to call Amazon Connect APIs for instance association.

8.  TranscriptProcessorConnectPolicy  (AWS::IAM::Policy)
    - IAM policy attached to the transcript processor.
    - Grants permissions specific to processing Amazon Connect audio
      and transcript data.

HOW THIS STACK CONTRIBUTES:
  Without this stack, Amazon Connect calls would never enter the LCA
  pipeline. This is the entry point — call starts in Connect →
  EventBridge fires → ContactEventProcessorFunction runs →
  call record created in LCA → agent sees the call in the UI.

STACK 3: LLMTEMPLATESTACK  (LCA-LLMTEMPLATESTACK-17VG7ZP9TO9VS)
================================================================================

PURPOSE:
  Manages the AI prompt templates that control how post-call summaries
  are generated. Stores prompts in DynamoDB so admins can customize
  summary format without redeploying the stack.

TOTAL RESOURCES: 5

--------------------------------------------------------------------------------
RESOURCE LIST & FUNCTIONALITY
--------------------------------------------------------------------------------

1.  LLMPromptTemplateTable  (AWS::DynamoDB::Table)
    - DynamoDB table that stores all LLM prompt templates.
    - Contains two key items:
        a) DefaultSummaryPromptTemplates — read-only, shipped with LCA.
           Contains the default instructions sent to the LLM for
           generating summaries (e.g., "Summarize the call, identify
           the issue, resolution, and next steps").
        b) CustomSummaryPromptTemplates — editable by admins.
           Any values here override the defaults. Preserved during
           stack updates so customizations aren't lost.
    - When a call ends, BedrockSummaryLambda reads this table to
      get the prompt before calling the LLM.

2.  LLMPromptUploadFunction  (AWS::Lambda::Function)
    - Runs once during deployment.
    - Uploads the default prompt templates into LLMPromptTemplateTable.
    - Ensures the table is populated with working defaults on first deploy.

3.  LLMPromptUploadFunctionLogGroup  (AWS::Logs::LogGroup)
    - CloudWatch log group for LLMPromptUploadFunction.
    - Captures logs from the template upload process.

4.  LLMPromptUploadRole  (AWS::IAM::Role)
    - IAM role for LLMPromptUploadFunction.
    - Permissions to write to DynamoDB (LLMPromptTemplateTable).

5.  LLMStorePromptTemplates  (AWS::CloudFormation::CustomResource)
    - CloudFormation custom resource that triggers LLMPromptUploadFunction
      during stack creation and updates.
    - Ensures templates are always up to date when the stack changes.

HOW THIS STACK CONTRIBUTES:
  Controls the quality and format of every post-call summary your agents
  see. The prompt template is the instruction given to the AI — it
  determines what the summary focuses on (issue, resolution, customer
  sentiment, action items, etc.). Admins can edit the custom prompts
  in DynamoDB console without any code changes or redeployment.

  Console Link to Edit Custom Prompts:
  https://us-east-1.console.aws.amazon.com/dynamodbv2/home?region=us-east-1
  #edit-item?itemMode=2&pk=CustomSummaryPromptTemplates&table=
  LCA-LLMTEMPLATESTACK-17VG7ZP9TO9VS-LLMPromptTemplateTable-KIPZX21ULBNQ

STACK 4: VPCSTACK  (LCA-VPCSTACK-FTQ6GQL2D5SQ)
================================================================================

PURPOSE:
  Creates the networking foundation — a Virtual Private Cloud (VPC) with
  public and private subnets. All Lambda functions and other services run
  inside this VPC for security and isolation.

TOTAL RESOURCES: 21

--------------------------------------------------------------------------------
RESOURCE LIST & FUNCTIONALITY
--------------------------------------------------------------------------------

1.  VPC  (AWS::EC2::VPC)
    - The isolated virtual network for the entire LCA stack.
    - All resources communicate privately within this network.
    - Isolated from other AWS accounts and the public internet
      (except through controlled gateways).

2.  VPCIGW  (AWS::EC2::InternetGateway)
    - Attaches to the VPC to allow resources in public subnets
      to communicate with the internet.
    - Required for NAT Gateways to route outbound traffic.

3.  VPCGateway  (AWS::EC2::VPCGatewayAttachment)
    - Attaches the Internet Gateway (VPCIGW) to the VPC.
    - Without this, the VPC has no internet connectivity at all.

4.  PublicSubnet1  (AWS::EC2::Subnet)
    - A subnet in Availability Zone 1 with internet access.
    - Hosts NAT Gateway 1.
    - Resources here can be reached from the internet (if configured).

5.  PublicSubnet2  (AWS::EC2::Subnet)
    - A subnet in Availability Zone 2 with internet access.
    - Hosts NAT Gateway 2.
    - Provides high availability across two AZs.

6.  PrivateSubnet1  (AWS::EC2::Subnet)
    - A subnet in Availability Zone 1 WITHOUT direct internet access.
    - Lambda functions run here for security.
    - Outbound internet access goes through NAT Gateway 1.

7.  PrivateSubnet2  (AWS::EC2::Subnet)
    - A subnet in Availability Zone 2 WITHOUT direct internet access.
    - Provides redundancy — if AZ1 fails, services continue in AZ2.

8.  PublicSubnet1NATGateway  (AWS::EC2::NatGateway)
    - NAT Gateway in AZ1 public subnet.
    - Allows Lambdas in PrivateSubnet1 to make outbound internet
      calls (e.g., calling Bedrock API, Transcribe API) without
      being directly exposed to the internet.

9.  PublicSubnet2NATGateway  (AWS::EC2::NatGateway)
    - NAT Gateway in AZ2 public subnet.
    - Same purpose as NATGateway1 but for PrivateSubnet2.
    - Provides redundancy.

10. NATGatewayEIP1  (AWS::EC2::EIP)
    - Elastic (static) IP address assigned to NAT Gateway 1.
    - All outbound traffic from private subnet 1 appears to come
      from this fixed IP address.

11. NATGatewayEIP2  (AWS::EC2::EIP)
    - Elastic IP for NAT Gateway 2.
    - Same purpose for private subnet 2.

12. PublicSubnet1RouteTable  (AWS::EC2::RouteTable)
    - Routing rules for PublicSubnet1.
    - Routes internet-bound traffic to the Internet Gateway.

13. PublicSubnet2RouteTable  (AWS::EC2::RouteTable)
    - Routing rules for PublicSubnet2.

14. PublicSubnet1DefaultRoute  (AWS::EC2::Route)
    - The default route (0.0.0.0/0) in PublicSubnet1's route table.
    - Sends all internet-bound traffic to the Internet Gateway.

15. PublicSubnet2DefaultRoute  (AWS::EC2::Route)
    - Default route for PublicSubnet2 to Internet Gateway.

16. PublicSubnet1RouteTableAssociation  (AWS::EC2::SubnetRouteTableAssociation)
    - Associates PublicSubnet1 with its route table.

17. PublicSubnet2RouteTableAssociation  (AWS::EC2::SubnetRouteTableAssociation)
    - Associates PublicSubnet2 with its route table.

18. VPCPrivateSubnet1RouteTable  (AWS::EC2::RouteTable)
    - Routing rules for PrivateSubnet1.
    - Routes internet-bound traffic to NAT Gateway 1 (not IGW directly).

19. VPCPrivateSubnet2RouteTable  (AWS::EC2::RouteTable)
    - Routing rules for PrivateSubnet2.

20. VPCPrivateSubnet1DefaultRoute  (AWS::EC2::Route)

21. VPCPrivateSubnet2DefaultRoute  (AWS::EC2::Route)

22. VPCPrivateSubnet1RouteTableAssociation  (AWS::EC2::SubnetRouteTableAssociation)

24. VPCFlowLog  (AWS::EC2::FlowLog)

HOW THIS STACK CONTRIBUTES:
  Provides the secure network backbone. Lambda functions (transcript
  processor, summary generators) run in private subnets — no direct
  internet exposure. NAT Gateways allow these Lambdas to call AWS
  services (Transcribe, Bedrock, DynamoDB) securely. The dual-AZ
  setup ensures the system stays up even if one availability zone
  has an outage.
