STACK 1: QNABOT — WHY IT IS SAFE TO DELETE
================================================================================

WHAT QNABOT DOES:
  QnABot is an agent assist Q&A system. When an agent types a question
  during a live call, QnABot searches its OpenSearch index and returns
  the most relevant answer. It uses Amazon Lex v2 for natural language
  understanding and has a Content Designer UI for admins to manage Q&A pairs.

WHY IT IS DISCONNECTED FROM YOUR PIPELINE:
  For AISTACK to use QnABot, it needs to know QnABot's Lambda function ARN.
  This ARN is passed via the parameter "AgentAssistExistingLambdaFunctionArn".

  Actual value in your deployment:
    AgentAssistExistingLambdaFunctionArn = (empty)

  Since this is empty, AISTACK has no way to call QnABot. It doesn't know
  QnABot exists. The CallEventProcessorFunction and
  AsyncTranscriptSummaryOrchestrator never invoke any QnABot Lambda.

  Additionally:
    IsLambdaAgentAssistEnabled = false
  This explicitly disables Lambda-based agent assist in AISTACK.

WHAT HAPPENS IF YOU DELETE IT:
  Nothing changes. Your agents will still see live transcripts, call
  summaries, and post-call summaries exactly as before. The only thing
  that disappears is the unused Content Designer UI and the OpenSearch
  Q&A search engine — neither of which your agents use.

COST SAVED:
  ~$150-300/month (primarily the always-on OpenSearch domain)

STACK 2: BEDROCKKB — WHY IT IS SAFE TO DELETE
================================================================================

WHAT BEDROCKKB DOES:
  BEDROCKKB creates an Amazon Bedrock Knowledge Base backed by an
  OpenSearch Serverless vector store. It crawls configured web URLs,
  converts content into vector embeddings, and stores them for semantic
  (meaning-based) search. This is used to answer agent questions during
  live calls using AI-powered knowledge retrieval.

IMPORTANT DISTINCTION — Two different uses of "Bedrock":
  There are TWO separate things in your stack that involve Bedrock:

  a) Amazon Bedrock AI Service (the LLM):
     - Used by BedrockSummaryLambda in AISTACK
     - Generates post-call summaries using Nova Lite model
     - Configured via: EndOfCallTranscriptSummary = BEDROCK
     - This is an AWS-managed service — NO stack needed, never deleted

  b) Bedrock Knowledge Base (the BEDROCKKB stack):
     - A vector search database for agent Q&A
     - Only used when agents ask questions during calls
     - Completely separate from post-call summary generation

  YOUR SUMMARIES USE (a) — the AI service directly.
  YOUR DEPLOYMENT NEVER USES (b) — the knowledge base stack.

WHY IT IS DISCONNECTED FROM YOUR PIPELINE:
  For BEDROCKKB to serve answers to agents, it must be connected through
  the AGENTASSISTSETUP stack, which in turn requires hook ARNs to be set
  in AISTACK. As shown above, all hook ARNs are empty. The chain is broken
  at the very first link — AISTACK never triggers agent assist at all.

WHAT HAPPENS IF YOU DELETE IT:
  Nothing changes. Post-call summaries continue to work because they use
  Amazon Bedrock the AI service (Nova Lite model) directly from within
  AISTACK — not the Knowledge Base. The OpenSearch Serverless vector
  collection simply gets removed, saving cost.

COST SAVED:
  ~$50-100/month (OpenSearch Serverless collection charges even when idle)

STACK 3: AGENTASSISTSETUP — WHY IT IS SAFE TO DELETE
================================================================================

WHAT AGENTASSISTSETUP DOES:
  This stack wires together QnABot and the Bedrock Knowledge Base with
  the LCA call pipeline. It contains Lambda functions that act as "hooks"
  — functions that get called at specific points during a call (start,
  during transcription, end) to provide agent assist suggestions and
  summaries through QnABot.

  Key Lambda functions in this stack:
    - QNASummarizeCallFunction    — summarizes calls via QnABot hook
    - QNABedrockKnowledgeBaseFunction — queries Bedrock KB for answers
    - SetupFunction               — wires everything together on deploy

WHY IT IS DISCONNECTED FROM YOUR PIPELINE:
  AGENTASSISTSETUP connects to AISTACK through 3 hook ARN parameters:

    Hook Parameter                        Your Value
    ------------------------------------ ----------------
    StartOfCallLambdaHookFunctionArn     (empty)
    TranscriptLambdaHookFunctionArn      (empty)
    PostCallSummaryLambdaHookFunctionArn (empty)
    EndOfCallLambdaHookFunctionArn       (empty)

  ALL FOUR ARE EMPTY.

  This means:
    - When a call STARTS   → AISTACK does NOT call AGENTASSISTSETUP
    - During transcription → AISTACK does NOT call AGENTASSISTSETUP
    - When a call ENDS     → AISTACK does NOT call AGENTASSISTSETUP
    - For summaries        → AISTACK goes DIRECTLY to Amazon Bedrock

  The QNASummarizeCallFunction in AGENTASSISTSETUP is never invoked.
  Your post-call summaries are generated entirely within AISTACK using
  BedrockSummaryLambda → Amazon Bedrock (Nova Lite) directly.

WHAT HAPPENS IF YOU DELETE IT:
  Nothing changes. The hook functions that were never called simply
  cease to exist. Your live transcript, call summary, and post-call
  summary features are completely unaffected.

COST SAVED:
  ~$0-5/month (Lambda is pay-per-use and was never being called anyway)

STACK 4: VPCSTACK — WHY IT IS SAFE TO DELETE
================================================================================

WHAT VPCSTACK DOES:
  Creates a Virtual Private Cloud (VPC) with public and private subnets,
  NAT Gateways, Internet Gateway, and flow logs. Provides a private,
  isolated network for services that handle raw audio streams and need
  network-level security.

WHY IT IS DISCONNECTED FROM YOUR PIPELINE:
  We checked the parent LCA CloudFormation template to see which stacks
  receive the VPC subnet IDs as parameters. Here is the complete list:

    Stack receiving VPC outputs          Deployed in your account?
    ------------------------------------ -------------------------
    CHIMEVCSTACK                         NO — not deployed
    GENESYSAUDIOHOOKTRANSCRIBERSTACK     NO — not deployed
    TALKDESKSTACK                        NO — not deployed
    WEBSOCKETTRANSCRIBERSTACK            NO — not deployed

    AISTACK                              YES — but does NOT use VPC
    CONNECTINTEGRATIONSTACK              YES — but does NOT use VPC
    LLMTEMPLATESTACK                     YES — but does NOT use VPC

  The VPC was designed for audio source stacks (Genesys, Talkdesk,
  WebSocket, Chime) that need private networking to handle raw audio.
  None of these are deployed in your environment.

  Your three active stacks (AISTACK, CONNECTINTEGRATIONSTACK,
  LLMTEMPLATESTACK) run their Lambda functions OUTSIDE the VPC, which
  is perfectly valid. AWS Lambda functions don't require a VPC — they
  can call Transcribe, Bedrock, DynamoDB, and AppSync directly over
  AWS's internal network without needing subnet configuration.

WHAT HAPPENS IF YOU DELETE IT:
  Nothing changes. No running service references any subnet or VPC ID
  from VPCSTACK. The 2 NAT Gateways and their Elastic IPs simply get
  removed, saving cost.

COST SAVED:
  ~$65-130/month (2 NAT Gateways at ~$32-65 each, charged hourly)
